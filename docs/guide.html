One-command Git Worktree lanes.

mtree Mekari Worktree

One command spins up an isolated working directory for a ticket — its own branch, its own dependencies, browsable in seconds — while every other lane keeps running. A few long-lived lanes, one shared .git, zero context-switch.

Built for your monorepo — not limited to it

Today mtree is optimized for a primary Web + Mobile-API monorepo, but the lane model is repo-agnostic — it works just as well for microservices repositories.

your-repo — zsh
$ mwt bugfix PROJ-1234 fix-export ▸ branch bugfix/PROJ-1234-fix-export off develop ▸ lane your-repo-worktrees/your-repo-bugfix-PROJ-1234 ▸ dependencies vendor + node_modules cloned (APFS) · .env copied ▸ url http://proj-1234.localhost ✓ ready in 3s

Why mtree

A monorepo checkout is a single desk. Reviewing a teammate's PR means stashing your half-done feature, switching branches, and waiting for Composer and Yarn to rebuild against a different lockfile — then doing it all again to switch back. git worktree already solves this by allowing multiple checkouts off one object store, but the raw plumbing is fiddly. mtree makes it a single command, the same way every time.

The lane model

Borrowed from matklad's concurrency-lane model: instead of one directory you keep thrashing, you keep a handful of long-lived working directories — each pinned to a role — all sharing one .git.

main The clean primary checkout on master — your source of truth. Never removed. persistent
mobile Standing lane off the mobile base, kept ready so you never re-provision it. persistent
work One per ticket, off develop. Run PROJ-1 and PROJ-2 side by side, fully isolated. per ticket
hotfix Branched off the production base for an urgent patch — without disturbing in-flight work. per ticket
review A detached lane parked at a PR's source branch. One branch in, read it, move on. detached
scratch Throwaway checkout in /tmp for a quick experiment you'll never keep. ephemeral

How worktrees are organized

Every lane mtree creates lives in one container directory next to your main checkout, named <repo>-<type>-<TICKET>-<desc>. The names are deterministic, so you always know which directory holds which branch. 

# ~/www/ — your workspace root
~/www/
├── your-repo/                                  ← main · branch: master
└── your-repo-worktrees/
    ├── your-repo-feature-PROJ-1234-export/  feature/PROJ-1234-export
    ├── your-repo-bugfix-PROJ-1305-login/      bugfix/PROJ-1305-login
    ├── your-repo-task-PROJ-1377-cleanup/       task/PROJ-1377-cleanup
    ├── your-repo-mobile/                        mobile-main
    └── your-repo-review/                        detached · one PR at a time

Browse any lane at its own URL

On creation, mtree registers a deterministic *.local hostname for each lane (an nginx vhost plus /etc/hosts and APP_URL). Every worktree is instantly browsable in isolation — sign in and pick a company exactly like production, without disturbing any other lane. 

# each lane gets its own *.local hostname — https://<ticket>-<repo>.local/
https://proj-1234-your-repo.local/      # → lane: bugfix/PROJ-1234
https://proj-1305-your-repo.local/      # → lane: feature/PROJ-1305

Setup · 3 Steps

From clone to your first lane

It's one Bash file with nothing to install. Drop it on your PATH and go.

1

Put mtree on your PATH

Clone the repository and symlink the script — that's the whole installation.

# macOS / zsh
git clone git@github.com:chan-mekari/mekari-worktree.git
ln -s "$PWD/mekari-worktree/mtree" /usr/local/bin/mtree
2

Run mtree init inside your project repository folder

Idempotent setup: it installs the mtree (REPL) and mwt (one-shot) aliases, creates the persistent Mobile-API lane, and wires up worktree includes. Safe to re-run any time. 

# run from the root of your project repository
mtree init
3

Open your first lane

Pick a type and a ticket. mtree creates the branch, installs dependencies, and hands you a ready-to-code directory with a live *.local URL. 

# → feature/PROJ-9000-reporting-export, dependencies ready
mwt feature PROJ-9000 reporting-export
Idempotent by design

Re-running mtree init never overwrites your existing configuration — it only adds what's missing. You can run it safely after pulling updates.

Daily Workflow

Ticket types are data, not code

Every type runs through one engine. The registry is a plain table near the top of the script — add a row and that type instantly works as both work <type> and a bare <type> subcommand. No new code path.

one row → one command

Built-in types

bugfix, task, story, feature, hotfix, and spike ship in the table. Each row names a prefix, its web and mobile base branches, and a mode — so the right branch always forks off the right base.

Run mwt types to print the live registry for your repository.

# TCW_TYPES — one row per type
# type   | prefix | base    | mode
bugfix   | bugfix | develop | standard
task     | task   | develop | standard
story    | story  | develop | standard
feature  | feature| develop | standard
hotfix   | hotfix | master  | standard
spike    | spike  | develop | standard

Dependencies are cloned, never symlinked

A fresh lane has no vendor/ or node_modules/. mtree fills them in the fastest safe way — and refuses the tempting shortcut that quietly corrupts your main checkout.

lockfile matches

Copy-on-write clone

When a lane's lockfile hash matches main, mtree does an APFS cp -c clone — isolated and near-instant, with no download. If it differs, it runs a real install. Composer always runs inside the laradock workspace container for PHP parity, and only .env is copied.

why not symlink?

Symlinks race and corrupt

Post-install hooks cp -rf into these directories, Yarn rewrites .yarn-integrity, and builds write node_modules/.cache. Through a symlink all of that would write straight back into your pristine main checkout — and two lanes would race on one directory. So mtree clones instead.

Common Commands

The commands you'll actually type

mtree opens an interactive REPL; mwt <cmd> runs one command and exits. Full list in mwt help.

start work
mwt <type> <TICKET> [desc]

Create the branch off its base and install dependencies. e.g. mwt bugfix PROJ-1 npe.

continue
mwt resume <branch>

Pick up an existing branch — local or origin — in its own lane.

urgent
mwt hotfix [web|mobile] <TICKET>

Branch off the production base for a patch, without touching in-flight lanes.

review
mwt review <branch>

Detached lane parked at a PR's source branch. Read it, then move on.

agent
mwt claude <TICKET> [args…]

Launch Claude Code inside a lane — isolated from everything else you're running.

inspect
mwt ls / mwt doctor

List every lane, or get a health report: disk, dependency drift, prunable / locked.

tidy
mwt remove <TICKET>

Drop a lane but keep the local branch. Never touches remotes.

tidy
mwt cleanup

Remove clean lanes already merged into the base. Skips locked or dirty ones.

Security

Working safely across lanes

Parallel worktrees are powerful, but they multiply the number of branches and environments you touch in a day. These practices keep your repository, your commits, and your local secrets safe.

Never commit directly to master

master is the production line. Always work in a lane (a feature/, bugfix/, or hotfix/ branch) and merge through a reviewed pull request.

Repository Safety

  • Never commit directly to master.
  • Keep each worktree isolated to a single branch.
  • Regularly sync your lanes from master.
  • Remove stale worktrees once their ticket ships.

Git Best Practices

  • Verify the current branch before every commit.
  • Avoid force-pushing shared branches.
  • Keep commits atomic and focused.
  • Rebase responsibly — only on branches you own.

Local Environment

  • Protect your SSH keys and never share them.
  • Never commit secrets or credentials.
  • Use environment variables for configuration.
  • Ignore local config files (mtree copies .env per lane).

Cleanup

  • Remove obsolete worktrees with mwt remove.
  • Clean unused branches once they're merged.
  • Verify worktree status before deletion.
  • Run mwt cleanup to sweep merged lanes.

mtree Commands

  • mwt remove keeps your local branch and never touches the remote.
  • mwt cleanup only removes lanes merged into base — it skips dirty or locked ones.
  • Run mwt doctor to audit lanes before any bulk cleanup.
  • Reach for --force only when you mean it (see below).

The mtree Script

  • It's one auditable Bash file — read it before adding it to your PATH.
  • Composer runs inside the laradock container, never on host PHP.
  • Only .env is copied into lanes (and it's gitignored) — secrets stay local.
  • Use --json for automation; human output goes to stderr, never mixed in.
--force is destructive

mwt remove <TICKET> --force discards uncommitted changes in that lane and unlocks a locked one. Run mwt ls or mwt doctor first so you never drop work you meant to keep.

Best practice — check before you delete

Run mwt doctor or mwt ls before removing a lane. mwt remove keeps your local branch and never touches the remote, so a clean removal is always reversible.

Remotes are never touched automatically

Lifecycle commands (remove, cleanup) only act on your local worktrees and branches. They never push, force-push, or delete anything on the remote.

Guide · Under the hood

APFS, reflinks & tuning

mtree's near-instant lane creation isn't magic — it's your file system. Here's the short version of how it works, and how to tune it per repository.

file system

What is APFS?

APFS (Apple File System) is the default file system on modern macOS. Its key trick is copy-on-write: when mtree copies vendor/ or node_modules/ into a new lane, APFS doesn't duplicate the bytes — it shares the original blocks and only writes the ones that actually change.

That's why cloning gigabytes of dependencies finishes in a second, and why ten lanes cost almost no extra disk until their files diverge — a big win in storage efficiency.

copy-on-write

What are reflinks?

A reflink (reference link) is a copy that points at the same underlying data blocks until one side is modified — the mechanism behind APFS cp -c. Supported on APFS (macOS) and Btrfs / XFS (Linux).

Because no bytes are read or written, copying a huge dependency tree is proportional to its metadata, not its size — so cloning is effectively free. If the file system doesn't support reflinks, mtree falls back to a normal recursive copy or a fresh install: correct, just slower.

Tuning — everything resolves per repository

resolution order  →  git config wt.<key>  ›  environment variable  ›  inferred default

git config wt.base develop
Pin the branch new lanes fork from, per repository.
git config wt.dockerDir …

--json · --base · --no-bootstrap

Troubleshooting

Start with mwt doctor

It reports disk usage, symlinked dependencies, .env drift, and prunable or locked lanes — most problems show up here first.

  • Dependencies out of date in a lane → re-run mwt bootstrap <path>.
  • Missing or stale .env → mwt sync-env <path> re-copies it from main.
  • A lane won't remove because it's locked → mwt remove <TICKET> --force.
  • Too many old lanes → mwt cleanup sweeps the ones already merged.

FAQ

Frequently asked questions

Does mtree ever touch my remotes?

No. Lifecycle commands like remove and cleanup only act on your local worktrees and branches — they never push, force-push, or delete on the remote.

Can I work on two tickets at the same time?

Yes — that's the point. Each lane is a separate working directory with its own branch and dependencies, so PROJ-1 and PROJ-2 run fully isolated and in parallel.

Why not just use git stash and git checkout?

Switching branches in one checkout reuses the same vendor/ and node_modules/, forcing a rebuild whenever lockfiles differ — and you lose in-flight state. Separate worktrees keep each branch's environment intact, so there's nothing to stash or rebuild.

Does mtree work outside of your-repo?

The core commands work in any Git repository. The init and mobile commands and the --web / --mobile platform split are your-repo-specific.

How do I add a new ticket type?

Add one row to the TCW_TYPES table near the top of the script. That type immediately works as both work <type> and a bare <type> subcommand — no other code changes needed. Once it works locally, commit your edit and open a pull request so the whole team gets the new type.

Is Linux or Windows supported?

mtree targets macOS with APFS today, where copy-on-write clones make lane creation near-instant. On file systems without reflink support it falls back to a regular copy or install — correct, just slower.

Collaborate

Contribute to mtree

mtree is one Bash file with no build step, so contributing is quick. Add a ticket type, fix a bug, or improve a command — then open a pull request for review.

1

Branch off main

Create a focused branch for your change.

# from your mekari-worktree clone
git switch -c add-review-type
2

Make your edit

To add a ticket type, add one row to the TCW_TYPES table near the top of the mtree script — that's all it takes for the type to work as both work <type> and a bare <type> command.

3

Sanity-check the script

There's no test suite — the one gate is a syntax check (plus an optional lint).

bash -n mtree        # syntax check (required)
shellcheck mtree     # optional, if installed
4

Commit, push, and open a pull request

Push your branch and open a PR against main. Keep commits focused, and describe what changed and why so it's easy to review.

git commit -am "Add review ticket type"
git push -u origin add-review-type
