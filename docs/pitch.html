A monorepo checkout is a single desk. Give every ticket, review, and AI agent its own clean desk — ready to run, in seconds.

The problem we pay for daily

Every PR review, hotfix, or parallel ticket in talenta-core means git stash + checkout + a minutes-long Composer/Yarn rebuild against a different lockfile — then all of it again to switch back. git worktree solves the plumbing, but raw, it's fiddly and easy to leave corrupt.

The solution — a lane for every job

A handful of long-lived working directories, each pinned to a role, all sharing one .git. Borrowed from matklad's concurrency-lane model; made one command, the same way every time.

mainThe clean primary checkout — your source of truth.persistent
mobileStanding lane off the Mobile-API base, kept ready.persistent
workOne per ticket — run TM-1 and TM-2 side by side, fully isolated.per ticket
hotfixOff the production base for an urgent patch, in-flight work untouched.per ticket
reviewDetached lane parked at a PR's source branch. Read it, move on.detached

What you actually type

zsh — talenta-core
mwt ❯ bugfix TM-1 npe # isolated branch + deps ready, in seconds
mwt ❯ review feature/X # read a teammate's PR, your work untouched
mwt ❯ claude TM-1 # launch an AI agent in its own clean lane
mwt ❯ cleanup # merged lanes self-garbage-collect

New — name it from the ticket, not from scratch

Paste a Jira URL and Claude reads the actual ticket — user story, acceptance criteria, description — then proposes a clean, convention-following branch name. You always approve or edit before anything is created. No embedded credentials: mtree delegates to the claude CLI you already use.

The one non-obvious idea

Dependencies are cloned, never symlinked.

When a lane's lockfile matches main, vendor/ and node_modules/ are materialized with an APFS copy-on-write reflink — near-instant, ~0 disk until they diverge, fully isolated.

✓ clone-on-write

Each lane gets its own real, isolated deps. Builds, caches, and two agents never touch each other.

✗ symlink (rejected)

One lane's build cache corrupts the pristine main checkout; two agents race a single directory.

This is the difference between parallel work that's safe and parallel work that occasionally eats your environment.

The ask

  • A maintainer owner — treat mtree as funded DX infrastructure, not a side project.
  • A lightweight smoke-test suite — none today; bash -n is the only gate.
  • Opt-in telemetry — so we measure the wins instead of asserting them.

Why now

  • Cost is low. One self-contained Bash file — no build, no deps, no server.
  • Risk is contained. A thin wrapper over git worktree; never pushes or deletes remotes.
  • AI agents make per-task isolation a throughput multiplier — and the tool already exists, in daily use.
