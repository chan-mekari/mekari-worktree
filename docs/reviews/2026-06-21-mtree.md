# Code Review — `mtree` (2026-06-21)

**File:** `/Users/chan/www/mekari-worktree/mtree`  
**Scope:** Whole file — refactoring + comment simplification  
**Stakes:** Developer tooling (used daily); not production-path code

## Summary

The script is in good shape overall. The logic is sound, the Docker + COW bootstrap rationale is well-documented, and the lane-lifecycle model is coherent. Three issues stand out: a misleading hotfix error message, a `type` name that shadows a bash builtin, and a block of duplicate output lines in `cmd_remove`. Beyond those, the main ask — comment simplification — surfaces roughly a dozen what-comments that should be cut or compressed.

---

## 🟠 Major

### 1. `cmd_hotfix` — error message blames the wrong argument (line 405)

```bash
# current
[[ -n "${2:-}" ]] && die "hotfix target must be 'web' or 'mobile', got: $1"
```

The `*)` branch fires when `$1` is a ticket (e.g. `TM-1`). If a second arg also appears, the die message prints `got: TM-1` — suggesting the *ticket* is invalid, not that a stray second argument was passed. A confused user would correct the ticket, not remove the extra arg.

**Fix:**
```bash
[[ -n "${2:-}" ]] && die "unexpected argument '$2' — usage: hotfix [web|mobile] <TICKET>"
```

## 🟡 Minor

### 2. `cmd_work` calls `_type_row` twice (lines 357, 362)

First call discards the row (used only as a bool check); second call retrieves it. The row fetch is cheap but the double-call is confusing — it looks like the type check and the data extraction are decoupled for a reason, but they aren't.

```bash
# current
if _type_row "$type" >/dev/null 2>&1; then shift; else type="feature"; fi
...
row="$(_type_row "$type")" || die "unknown ticket type..."
```

**Fix:** Collapse into one call:
```bash
if ! row="$(_type_row "${1:-}")"; then
  type="feature"; row="$(_type_row "feature")"
else
  type="$1"; shift
fi
```

### 3. `type` shadows the bash builtin (line 356, also `cmd_hotfix` line 400)

`local type=...` and `type=hotfix` shadow `type`, a bash built-in that checks if a command exists. The script never calls the builtin inside these functions, so it doesn't currently break — but it's a trap for anyone debugging with `type cmd_work` inside these scopes.

**Fix:** Rename to `ticket_type` (or `wtype`) in both `cmd_work` and `cmd_hotfix`.

---

### 4. `cmd_remove` duplicates its summary output block

The summary lines (Ticket / Path / Branch) appear almost identically in two paths: the orphaned-entry fast-path (lines 629–631) and the normal removal path (lines 665–670). The orphan path omits `ok "Registry updated"` — the divergence is easy to miss on future edits.

**Fix:** Extract a `_remove_summary` helper:
```bash
_remove_summary() {  # <ticket> <path> <branch> <status>
  ok "Worktree removed"; [[ "$4" != "pruned" ]] && ok "Registry updated"
  _say "Ticket: $1"; _say "Path:   $2"; _say "Branch: $3 (preserved)"
  _emit "$3" "" "$2" "$4"
}
```

---

### 5. `_t` throwaway in `cmd_work` IFS-read (line 362)

`IFS='|' read -r _t prefix web_base mobile_base mode <<<"$row"` uses `_t` as a discard for the first field (which duplicates `$type`). The `_` convention (`read -r _ prefix …`) is more idiomatic and signals intent.

### 6. Stale doc reference in `cmd_init` (line 673)

`# Idempotent setup (Confluence §3): ...` — the `(Confluence §3)` callout is a dead link; the page or section number has likely drifted. External doc references in inline comments rot silently.

**Fix:** Remove the Confluence callout; the sentence after it is sufficient.
